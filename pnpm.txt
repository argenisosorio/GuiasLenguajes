===========
npm vs pnpm
===========

Si alguna vez has sentido que tu carpeta node_modules pesa más que tu propia
alma, entender la diferencia entre npm y pnpm te va a cambiar la vida. Ambos son
gestores de paquetes para Node.js, pero funcionan bajo filosofías muy distintas.

1. La diferencia fundamental: El almacenamiento

La mayor diferencia radica en cómo guardan los paquetes en tu disco duro.

-npm (Node Package Manager): Es el estándar por defecto. Si tienes 10 proyectos
que usan lodash, npm descargará y guardará 10 copias de lodash, una en cada
carpeta node_modules. Es ineficiente con el espacio.

pnpm (Performant npm): Utiliza un almacén de contenido direccionable
(Content-addressable store). Si 10 proyectos usan lodash, pnpm solo guarda una
copia en una carpeta global de tu disco y crea "enlaces" (hard links) hacia
ella.

Resultado: Ahorras gigabytes de espacio y las instalaciones son ridículamente
rápidas.

2. Velocidad y Rendimiento

En la mayoría de las comparativas, pnpm gana por goleada.

+----------------------+-------------------------------+----------------------------------+
| Característica       | npm                           | pnpm                             |
+----------------------+-------------------------------+----------------------------------+
| Instalación inicial  | Lenta (descarga todo de nuevo)| Muy rápida (usa el caché global) |
+----------------------+-------------------------------+----------------------------------+
| Uso de disco         | Muy alto                      | Muy bajo (enlaces simbólicos)    |
+----------------------+-------------------------------+----------------------------------+
| Seguridad            | Buena                         | Excelente (evita dep. fantasma)  |
+----------------------+-------------------------------+----------------------------------+
| Madurez              | El estándar de la industria   | Cada vez más usado               |
+----------------------+-------------------------------+----------------------------------+

3. El problema de las "Dependencias Fantasma"

npm utiliza una estructura plana (flattened). Esto significa que si instalas el
paquete A, y este depende del paquete B, npm pone a ambos en la raíz de
node_modules. Esto te permite usar el paquete B en tu código aunque no lo hayas
instalado tú directamente.

pnpm no permite esto. Utiliza enlaces simbólicos para crear una estructura
estricta. Solo puedes importar lo que declaraste explícitamente en tu
package.json. Esto evita errores misteriosos en producción.

4. Comandos básicos

La buena noticia es que la curva de aprendizaje es casi nula porque los comandos
son prácticamente idénticos:

Instalar dependencias:

-npm: npm install

-pnpm: pnpm install

Añadir un paquete:

-npm: npm install express

-pnpm: pnpm add express

Ejecutar un script:

-npm: npm run dev

-pnpm: pnpm dev (ni siquiera necesitas el "run")

¿Cuál deberías elegir?

Usa npm si: Eres principiante, quieres evitar configurar herramientas
adicionales o trabajas en un equipo que no quiere cambiar su flujo de trabajo
actual.

Usa pnpm si: Tienes poco espacio en disco, trabajas en monorepos (donde pnpm
brilla especialmente) o simplemente odias esperar minutos a que termine un
install.

Ejemplo práctico: Un "Hola Mundo" con el paquete confeti
========================================================

Para notar la ventaja real de pnpm, lo ideal es ver cómo maneja el espacio en
disco comparado con npm. El "truco" está en que pnpm no duplica archivos; si dos
proyectos usan la misma librería, solo ocupa espacio una vez.

Aquí tienes un ejercicio práctico para que lo compruebes tú mismo:

Paso 1: Instalación (si no lo tienes)

Si tienes Node.js instalado, puedes activar pnpm rápidamente:

$ corepack enable

O vía npm:

$ npm install -g pnpm

Verificar la versión instalada:

$ pnpm -v

Paso 2: Crear el Proyecto A

Para notar la ventaja real de pnpm, vamos a usar canvas-confetti, un paquete que
añade una animación festiva a tu index.html.

$ mkdir proyecto-confeti-a && cd proyecto-confeti-a

$ pnpm init

$ pnpm add canvas-confetti

El comando anterior va a fallar con un error como este:

Invalid package manager specification in package.json (pnpm@^11.1.1); expected a
semver version

El error ocurre porque la especificación packageManager (y en este caso, la
extensión devEngines) no acepta rangos de versiones como ^11.1.1. Este campo
requiere una versión exacta (un string fijo) para asegurar que todos los
desarrolladores usen la misma versión del gestor de paquetes.

Para solucionarlo, debes editar tu package.json y eliminar el símbolo ^.

Modifica la sección de devEngines (o packageManager si la usaras directamente)
para que quede así:

{
  "name": "proyecto-confeti-a",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "keywords": [],
  "author": "",
  "license": "ISC",
  "devEngines": {
    "packageManager": {
      "name": "pnpm",
      "version": "11.1.1",
      "onFail": "download"
    }
  },
  "type": "module"
}

Ejecuta de nuevo este comando para que se instale el paquete:

$ pnpm add canvas-confetti

Esto habrá creado la carpeta node_modules y el archivo pnpm-lock.yaml

Ahora, crea un archivo index.html básico:

<!DOCTYPE html>
<html>
<body>
  <h1>¡Hola Mundo con pnpm!</h1>
  <script type="module">
    import confetti from './node_modules/canvas-confetti/dist/confetti.module.mjs';
    confetti();
  </script>
</body>
</html>

En este momento, pnpm descargó el paquete a una carpeta global y creó un enlace
en tu proyecto.

Paso 3: Probar el index.html

Usaremos un servidor local, para que el navegador trate tu proyecto como un
sitio web real, necesitas servirlo a través de un servidor local. Como ya tienes
pnpm y Node.js, la forma más rápida es usar un paquete llamado serve o
live-server.

Ejecuta este comando en tu terminal (dentro de proyecto-a):

$ pnpm dlx serve .

Al abrir http://localhost:3000/ deberías ver el Hola mundo y una animación de
confeti.

¿Qué acaba de pasar?
====================

-Es instantáneo: La instalación en el segundo proyecto toma milisegundos porque
el código ya está en el almacén global de tu disco.

-No ocupa espacio extra: Aunque veas la carpeta node_modules, no hay una segunda
copia física del paquete. Ambos proyectos están "viendo" el mismo archivo
centralizado.

¿Cómo verificar la ventaja de espacio? Puedes ver dónde guarda pnpm los archivos
realmente con:

$ pnpm store path

Tus carpetas node_modules son técnicamente "espejos" (enlaces simbólicos) que
apuntan a esa ruta única.

Revisión de la estructura:

$ tree

├── index.html
├── node_modules
│   └── canvas-confetti -> .pnpm/canvas-confetti@1.9.4/node_modules/canvas-confetti
├── package.json
└── pnpm-lock.yaml

Esa salida del comando tree es la prueba perfecta de cómo funciona pnpm por
debajo. Fíjate en la flecha (->):

canvas-confetti -> .pnpm/canvas-confetti@1.9.4/node_modules/canvas-confetti

Esto confirma que canvas-confetti en tu carpeta node_modules no es una carpeta
real con archivos dentro, sino un enlace simbólico (symlink).

Resumen del beneficio:
======================

-Velocidad: El Proyecto B se configuró sin descargar nada de internet.

-Espacio: Si haces 50 pequeños experimentos con confeti, solo ocupas el espacio
de 1.

-Seguridad: Tienes la certeza de que el paquete es idéntico en todos tus
proyectos.

========================
Errores con node_modules
========================


Antes se usaba el viejo "borrón y cuenta nueva" de npm borrando la carpeta
node_modules. Sin embargo, la gran ventaja de haber migrado el entorno a pnpm es
que maneja las dependencias mediante un almacén global indexado
(el virtual store) y enlaces simbólicos (symlinks). Esto significa que los
problemas de corrupción de paquetes son muchísimo menos comunes, pero cuando
ocurren, la forma de solucionarlos cambia un poco.

1. La alternativa directa al clásico rm -rf node_modules

Si sospechas que los enlaces simbólicos de tu proyecto local se rompieron o
quedaron corruptos, puedes recrearlos en un segundo sin tener que volver a
descargar nada de internet.

Ejecuta en la raíz de tu proyecto:

$ pnpm store prune

$ pnpm install --force

pnpm store prune: Elimina del almacén global todos los paquetes huérfanos que ya
no se usan en ningún proyecto, liberando espacio.

--force: Le dice a pnpm que reconstruya la carpeta node_modules desde cero,
verificando y reemplazando todos los archivos modificados o corruptos utilizando
el caché global. ¡Es increíblemente rápido comparado con npm!

2. El reinicio "Hard" (El equivalente exacto a tu vieja costumbre)

Si quieres replicar exactamente el comportamiento de borrar todo y reinstalar
(por ejemplo, si cambiaste drásticamente de rama o sospechas del archivo de
bloqueo), puedes limpiar el directorio local y reinstalar de forma limpia:

En Linux/Debian:

$ rm -rf node_modules pnpm-lock.yaml && pnpm install

Nota: Al borrar el pnpm-lock.yaml, forzarás a pnpm a recalcular el árbol de
dependencias completo en su versión compatible más reciente según tu
package.json.

3. Verificar la integridad del almacén global

A veces, el problema no está en tu proyecto, sino en que algún archivo dentro
del almacén global de pnpm se corrompió (por ejemplo, si un proceso se
interrumpió abruptamente).

Para verificar si todos los paquetes del caché coinciden exactamente con sus
sumas de verificación (checksums) originales, usa:

$ pnpm store status

Si el comando detecta que algún paquete fue modificado externamente o está
corrupto, te avisará para que puedas correr un pnpm install --force y repararlo.
